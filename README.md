# Agentic Knowledge Compiler (AKC)

**Turn messy real-world information into executable knowledge.**

The Agentic Knowledge Compiler takes heterogeneous inputs—documents, messaging threads, API specs—and compiles them into runnable artifacts: code, workflows, and agent specifications. It emphasizes **compilation** (not just summarization), **living alignment** to sources, and **correctness-aware** design (tests, repair loops, optional verification).

## What’s the problem?

- Real-world knowledge lives in docs, Slack, APIs, and tickets—scattered and unstructured.
- One-shot codegen produces brittle outputs that drift from sources.
- Agent systems often behave as black boxes with no clear correctness story.

## What does AKC do?

| Others do                | AKC does                                                                              |
| ------------------------ | ------------------------------------------------------------------------------------- |
| Summarization (RAG, Q&A) | **Compilation** → runnable code, workflows, and agent specs                           |
| One-shot codegen         | **Living systems** that stay aligned to sources via re-ingestion and repair           |
| Black-box agents         | **Correctness-aware** design: tests, synthesize–execute–repair, optional verification |

## Quick start

**Requirements:** Python 3.11+, [uv](https://docs.astral.sh/uv/) (recommended) or pip.

```bash
# Clone and install
git clone https://github.com/your-org/agentic-knowledge-compiler.git
cd agentic-knowledge-compiler
uv sync
uv sync --extra dev   # for development (pytest, ruff, mypy)

# Run the CLI
uv run akc
```

**Reproducible one-liner (ingest → compile → verify):** Default flows are offline-first; no API keys required. Cloud embedders (OpenAI, Gemini) and optional index backends (sqlite, pgvector) are opt-in.

```bash
uv sync && uv run akc ingest --tenant-id my-tenant --connector docs --input ./docs --embedder hash --index-backend memory --no-index
uv run akc compile --tenant-id my-tenant --repo-id my-repo --outputs-root ./out
uv run akc verify --tenant-id my-tenant --repo-id my-repo --outputs-root ./out
```

Compile alone (assuming ingestion was run earlier or you use in-memory state):

```bash
uv run akc compile --tenant-id my-tenant --repo-id my-repo --outputs-root ./out
```

Artifacts (manifest, tests, code memory) live under `./out/<tenant-id>/<repo-id>/`.

## Compile loop capabilities and assumptions

- **Capabilities**
  - **Tiered controller:** ARCS-style small → medium → large tiers for generate/repair, with escalation on failures and clear accounting of tier history.
  - **Tests by default:** Every candidate runs tests, with support for `full` or `smoke`+`full` modes, periodic full runs, and promotion gates that require a passing full test stage.
  - **Policy gates:** Enforces a “tests generated by default” policy (non-test changes must be accompanied by tests) and an optional verifier gate that can veto unsafe or cross-tenant patches.
  - **Tenant + repo isolation:** All LLM and executor calls are scoped by `(tenant_id, repo_id)` and patches are treated as artifacts; applying them to a working tree is left to a higher-level, tenant-safe executor.
  - **Code memory integration:** Successful patches and their test outputs are persisted into code memory as first-class items, keyed by plan and step, for later retrieval and drift analysis.
- **Assumptions**
  - **Unified diff patches:** LLM backends are expected to return deterministic unified diff patches (no prose or markdown fences) so the controller can extract touched paths and test files.
  - **Test layout:** Test discovery assumes conventional Python layouts (e.g. `tests/`, `test_*.py`, `*_test.py`) when enforcing the “tests generated by default” policy.
  - **Executor sandboxing:** The executor is responsible for sandboxing, resource limits, and enforcing tenant isolation at the process/filesystem level; the controller only passes scoped commands.
  - **Budget configuration:** Callers must provide a `ControllerConfig` with sane budgets (LLM calls, max repairs, wall-clock limits) appropriate for their environment; the controller treats these as hard limits.

## Development and CI

- **Lint:** `uv run ruff check .`
- **Format:** `uv run ruff format .`
- **Types:** `uv run mypy src/akc`
- **Tests:** `uv run pytest`

CI runs these on every push and PR. Pre-commit hooks: `uv run pre-commit install`.

### Releases

- **Release automation:** Versions and tags are managed by `python-semantic-release` from conventional commit history.
- **Release flow:**
  - Merge feature branches into `dev`.
  - Promote `dev` into `release/*` and then into `main`.
  - On pushes to `main` or `release/*`, the `Release` workflow (`.github/workflows/release.yml`) runs:
    - Calculates the next semantic version from commits.
    - Updates project metadata and changelog.
    - Creates a Git tag and GitHub Release.

### Branch flow and CI gates

- **Branch model:**
  - Feature branches → `dev`
  - `dev` → `release/*` (promotion branches)
  - `release/*` → `main`
- **CI workflow:** `.github/workflows/ci.yml`
  - Triggers on `push` and `pull_request` for `dev`, `release/*`, and `main`.
  - Jobs:
    - `ci`: lint, format check, typecheck, tests (required everywhere).
    - `dafny` / `verus`: optional formal verification when Dafny/Verus modules are present or the PR is labeled `correctness`.
